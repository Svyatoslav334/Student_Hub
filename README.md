# StudentHub

StudentHub — університетський портал для студентів, викладачів та адміністрації. Він об'єднує новини, навчальні матеріали, студентські гуртки, документообіг, FAQ, чати в реальному часі та інтерактивну карту навчального корпусу в єдиному середовищі.

Проєкт складається з двох частин: фронтенду на React/TypeScript та бекенду на NestJS з базою даних PostgreSQL.

## Можливості

### Для студентів

- Стрічка новин з категоріями, закріпленими публікаціями та коментарями
- Каталог навчальних матеріалів (лекції, презентації, методички) з фільтрацією за категоріями та типами
- Студентські гуртки: перегляд, запис і вихід, чат учасників
- Чат академічної групи з куратором
- База викладачів із контактами, кабінетами та годинами консультацій
- Розділ документів — шаблони, бланки, зразки
- FAQ з можливістю поставити власне питання
- Інтерактивна карта корпусу з побудовою маршруту між аудиторіями

### Для викладачів

- Особиста панель зі статистикою
- Керування власними навчальними матеріалами
- Створення та модерація студентських гуртків, чат із учасниками
- Редагування власного викладацького профілю

### Для адміністраторів

- Дашборд із загальною статистикою системи
- Керування користувачами та їхніми ролями
- Керування студентськими групами та кураторами
- Модерація новин, FAQ та документів
- Редактор карти корпусу — малювання приміщень, коридорів і стін

## Технологічний стек

### Frontend

- React 18 + TypeScript
- Vite
- React Router
- Zustand (стейт автентифікації)
- Tailwind CSS
- Axios
- Socket.IO Client
- Lucide React (іконки)
- Sonner (сповіщення)

### Backend

- NestJS
- TypeORM + PostgreSQL (Supabase)
- Passport + JWT
- Socket.IO (WebSocket Gateway)
- Multer + Supabase Storage (завантаження файлів)
- class-validator / class-transformer

## Ролі користувачів

| Роль    | Опис                                                              |
|---------|-------------------------------------------------------------------|
| STUDENT | Базова роль, доступна одразу після реєстрації                    |
| TEACHER | Викладач: окремий профіль, керування матеріалами та гуртками      |
| ADMIN   | Повний доступ до адмін-панелі та керування системою               |

Доступ до сторінок на фронтенді обмежується компонентом `PrivateRoute`, а до маршрутів API — `RolesGuard` на бекенді.

## Модулі бекенду

| Модуль          | Призначення                                                            |
|------------------|-------------------------------------------------------------------------|
| `auth`           | Реєстрація, вхід та видача JWT-токенів                                  |
| `users`          | Список користувачів, зміна ролей, прив'язка до груп                    |
| `profile`        | Особистий профіль, аватар, соцмережі, біографія                        |
| `news`           | Новини з категоріями, закріпленням та коментарями                       |
| `materials`      | Навчальні матеріали з категоріями та прикріпленими файлами              |
| `clubs`          | Гуртки, учасники та чат у реальному часі                                |
| `student-groups` | Академічні групи, куратори та чат групи                                 |
| `teachers`       | Картки викладачів та їхні профілі                                       |
| `faq`            | Питання та відповіді, питання від студентів                             |
| `documents`      | Документи, шаблони та категорії документів                              |
| `campus-map`     | Карти поверхів, приміщення та маршрути                                  |

Файли користувачів (аватари, зображення новин і гуртків, матеріали, документи) зберігаються в Supabase Storage.

## Реальний час (WebSocket)

Чат гуртків і чат академічних груп працюють через Socket.IO:

- `joinClubRoom` / `sendMessage` — чат гуртка
- `joinGroupRoom` / `sendGroupMessage` — чат групи

Підключення авторизується за JWT-токеном, який передається у `handshake.auth.token`. Доступ до кімнати чату перевіряється на сервері — учасником гуртка/групи, куратором або керівником гуртка.

## Карта корпусу

Карта складається з приміщень і коридорів, які адміністратор малює у вбудованому редакторі. На основі цих даних будується граф з'єднань між приміщеннями, а маршрут між двома аудиторіями прокладається алгоритмом A* з подальшим спрощенням ламаної лінії алгоритмом Рамера–Дугласа–Пекера.

Кожен поверх зберігається як окремий запис із SVG-розміткою приміщень і списком кімнат із координатами та дверними проходами.

## Деплой на Render

Проєкт можна задеплоїти на [Render](https://render.com) як два окремих сервіси: backend (Web Service) та frontend (Static Site).

### Backend (Web Service)

1. У Render натиснути **New → Web Service**.
2. Підключити репозиторій проєкту та обрати гілку (`main`).
3. Вказати кореневу директорію (Root Directory): `backend`.
4. Налаштування збірки:
   - **Build Command**: `npm install && npm run build`
   - **Start Command**: `npm run start:prod`
   - **Environment**: `Node`
5. Додати змінні середовища (Environment → Add Environment Variable):

   ```env
   DB_HOST=
   DB_PORT=
   DB_USER=
   DB_PASSWORD=
   DB_NAME=

   JWT_SECRET=

   SUPABASE_URL=
   SUPABASE_SERVICE_KEY=

   PORT=10000
   FRONTEND_URL=https://<назва-фронтенду>.onrender.com
   ```

   > Render надає змінну `PORT` автоматично, але для NestJS її варто явно прописати (наприклад, `10000`) та переконатися, що застосунок слухає `process.env.PORT`.

6. Якщо база даних — Supabase, дані для `DB_HOST`, `DB_USER`, `DB_PASSWORD`, `DB_NAME` беруться з налаштувань підключення Supabase (Connection Pooling / Direct connection).
7. Після натискання **Create Web Service** Render автоматично виконує білд і деплой. URL бекенду матиме вигляд `https://<назва-бекенду>.onrender.com`.

### Frontend (Static Site)

1. Натиснути **New → Static Site**.
2. Підключити той самий репозиторій.
3. Вказати кореневу директорію: `frontend`.
4. Налаштування збірки:
   - **Build Command**: `npm install && npm run build`
   - **Publish Directory**: `dist`
5. Додати змінну середовища:

   ```env
   VITE_API_URL=https://<назва-бекенду>.onrender.com
   ```

6. Для коректної роботи React Router потрібно додати правило перенаправлення (Rewrite Rule) у налаштуваннях Static Site:
   - **Source**: `/*`
   - **Destination**: `/index.html`
   - **Action**: `Rewrite`

7. Після натискання **Create Static Site** та завершення білду сайт стає доступним за адресою `https://<назва-фронтенду>.onrender.com`.

### Після деплою

- `FRONTEND_URL` у налаштуваннях бекенду оновлюється на реальний URL фронтенду (для коректної роботи CORS).
- `VITE_API_URL` у налаштуваннях фронтенду оновлюється на реальний URL бекенду, після чого білд перезапускається.
- WebSocket-з'єднання (Socket.IO) мають проходити через HTTPS/WSS — Render підтримує це з коробки для Web Service.
- Безкоштовний план Render (Free) "засинає" після періоду неактивності — перший запит після сну може займати до 30-60 секунд.
